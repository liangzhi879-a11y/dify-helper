# Skill: Strict Code Review

按以下 checklist 逐项审查，**每发现一个问题必须明确指出文件+行号+问题描述+修复建议**。

## Checklist

### 正确性
- [ ] 逻辑是否符合需求？
- [ ] 边界条件处理：空值/零值/负值/极大值/空集合/单元素
- [ ] 循环终止条件正确？off-by-one？
- [ ] 类型转换是否安全？精度丢失？
- [ ] 字符串编码：UTF-8？多语言字符？

### 异常处理
- [ ] 外部调用（IO/网络/DB/API）都有 try-catch？
- [ ] 异常是否被吞掉？日志是否充分？
- [ ] 资源释放：文件/连接/锁 是否在 finally 或 context manager 中？
- [ ] 错误码/错误信息是否清晰可定位？

### 并发
- [ ] 共享状态是否有竞态？
- [ ] 锁的粒度是否合理？是否有死锁风险？
- [ ] 异步代码是否正确 await？是否有未捕获的 coroutine？
- [ ] 闭包捕获循环变量是否安全？

### 安全
- [ ] 外部输入是否校验？SQL 注入/XSS/路径遍历/命令注入？
- [ ] 敏感信息是否泄漏到日志/响应/错误信息？
- [ ] 权限校验是否到位？越权访问？
- [ ] 加密：密码/Token 是否安全存储（bcrypt/哈希）？

### 可维护性
- [ ] 函数职责单一？过长（>50 行）需要拆分？
- [ ] 命名是否清晰？是否需要注释解释 why？
- [ ] 重复代码？可否抽象？
- [ ] 魔法数字？应提取为常量？
- [ ] 圈复杂度过高？嵌套超过 4 层？

### 性能
- [ ] N+1 查询？
- [ ] 不必要的循环嵌套？O(n²) 可优化为 O(n)？
- [ ] 大对象是否懒加载？
- [ ] 缓存使用合理？过期策略？
- [ ] 不必要的序列化/反序列化？

### API 设计
- [ ] 接口语义清晰？RESTful 规范？
- [ ] 错误响应统一格式？
- [ ] 版本兼容？破坏性变更？
- [ ] 限流/熔断？

### 测试
- [ ] 是否有单元测试？覆盖率？
- [ ] 边界用例覆盖？
- [ ] Mock 是否合理？是否过度 Mock 导致测试无意义？
- [ ] 测试是否独立可重复？

## 输出格式

按严重程度分级：

```
🔴 严重（必须修复）：
1. [file.py:42] 问题描述 → 修复建议
2. ...

🟡 警告（建议修复）：
1. [file.py:88] 问题描述 → 修复建议
2. ...

🔵 建议（可选优化）：
1. [file.py:120] 问题描述 → 修复建议
2. ...

## 总结
- 严重问题：<N>
- 警告：<N>
- 建议：<N>
- 整体评价：<一段话>
```

## 禁止行为

- 禁止只说"看起来不错"无具体反馈
- 禁止不指出文件和行号
- 禁止只批评不给修复建议
- 禁止对风格问题过度苛责（除非团队有规范）
- 禁止忽略安全问题
