# Skill: Systematic Thinking

面对任何非平凡的开发任务，**禁止直接开始编码**，必须先完成以下步骤。简单任务（修个 typo、改个常量）可省略方案对比，但前两步必须完成。

## 1. 复述理解
用自己的话复述用户需求，确认无误后再继续。如有歧义，先提问澄清。

## 2. 拆解任务
将任务拆为可独立验证的子任务，每个子任务输出预期结果。

## 3. 方案对比
至少考虑 2 种实现方案，列出 trade-off（复杂度/性能/可维护性/可扩展性），推荐其一并说明理由。

## 4. 影响面分析
改动会影响哪些模块？是否破坏现有接口？是否需要数据迁移？是否有兼容性风险？

## 5. 验证策略
如何验证实现正确？需要哪些测试？如何手动验证？回滚方案？

## 输出格式（实施前）

```
【理解】<复述需求>
【拆解】
1. <子任务1> → 预期：<结果>
2. <子任务2> → 预期：<结果>
【方案】
- 方案 A: <描述> 优势：<...> 劣势：<...>
- 方案 B: <描述> 优势：<...> 劣势：<...>
推荐：<A/B>，理由：<...>
【影响】<受影响模块 + 兼容性>
【验证】<测试方法 + 验证步骤>
```

得到用户确认后再进入实施。

## 禁止行为

- 禁止跳过"复述理解"直接编码
- 禁止只考虑一种方案就开干（除非简单任务）
- 禁止忽略影响面分析
- 禁止没有验证策略就交付

## 适用场景

- 创建新功能/模块
- 修改现有核心逻辑
- 重构/迁移
- 涉及多文件的改动
- 涉及外部系统/接口的改动
- 不确定如何下手的需求

## 不适用场景（可直接编码）

- 修 typo / 改字符串
- 调整常量值
- 添加注释
- 格式化代码
- 单一文件小改动且无歧义
