# Skill: Refactor Patterns

识别代码坏味道并应用合适的设计模式重构。**不过度工程化**——能简单就别复杂。

## 识别坏味道

| 坏味道 | 症状 | 重构手法 |
|---|---|---|
| 重复代码 | 多处相同/相似的代码块 | 提取函数 / 提取超类 / 模板方法 |
| 函数过长 | >50 行，圈复杂度高 | 提取函数 / 用查询替换临时变量 |
| 类过大 | 单类职责过多 | 提取类 / 移动方法 |
| 参数列表过长 | >4 个参数 | 引入参数对象 / 保留对象 |
| 散弹式修改 | 一个改动改多处 | 移动函数 / 合并类 |
| 发散式变化 | 一个类因不同原因改 | 提取类 |
| 数据泥团 | 多处出现相同的字段组合 | 提取类 |
| 基本类型偏执 | 用 int/string 表示概念 | 值对象 / 类型别名 |
| switch 语句 | 大量 switch/if-else | 多态 / 策略模式 |
| 平行继承体系 | 加子类必须加另一个 | 移动方法 / 合并继承 |
| 冗赘类 | 类没用 | 内联类 / 删除 |
| 夸夸其谈 | 为未来设计 | 删除 / 延迟到需要时 |
| 临时字段 | 字段只在某些情况下用 | 提取类 / 设为 null |
| 数据类 | 只有 getter/setter | 移动行为到类 |
| 被拒绝的馈赠 | 子类不需要父类方法 | 用组合替换继承 |
| 注释过多 | 注释解释烂代码 | 重构让代码自解释 |

## 常用设计模式（按场景选）

### 创建型
- **工厂方法**：创建对象逻辑复杂，或需根据类型创建
- **抽象工厂**：创建一组相关对象
- **单例**：全局唯一（谨慎用，多数场景可用模块级变量替代）
- **建造者**：复杂对象分步构建（参数多时）

### 结构型
- **适配器**：接口不兼容时转换
- **装饰器**：动态添加功能，替代继承
- **外观**：为复杂子系统提供简单接口
- **组合**：树形结构统一处理
- **代理**：延迟加载/访问控制

### 行为型
- **策略**：算法族可互换（替代 switch）
- **观察者**：发布订阅解耦
- **命令**：请求封装为对象（撤销/队列/日志）
- **模板方法**：定义骨架，子类实现细节
- **状态**：状态机用类替代 if-else
- **责任链**：多个处理器依次尝试

## 重构步骤

1. **建立测试基线**：重构前必须有可运行的测试，确保行为不变
2. **小步重构**：每一步都保持代码可运行、测试通过
3. **频繁提交**：每完成一个小重构就 commit，便于回滚
4. **单一意图**：一次只做一种重构，不要混杂

## 何时不要重构

- 没有测试覆盖
- 接近 deadline
- 重构成本 > 收益
- 代码即将废弃
- 只为"看起来更好"没有实际收益

## 输出格式

```
【坏味道】<识别出的具体问题>
【位置】<file:line>
【影响】<可维护性/性能/可读性>
【重构方案】<具体手法 + 目标结构>
【步骤】
1. <小步骤1>
2. <小步骤2>
3. ...
【验证】<跑测试确保行为不变>
```

## 禁止行为

- 禁止大爆炸式重构（一次改一堆）
- 禁止重构 + 修 bug 同时进行
- 禁止为重构而重构（必须有明确收益）
- 禁止引入过度抽象（YAGNI）
- 禁止跳过测试直接重构
