# Skill: Test-First Thinking

写功能代码前先想清楚测试。不必严格 TDD（先写测试再写实现），但必须先想清楚"如何验证这个功能正确"。

## 思考顺序

### 1. 接口设计先行
明确函数/类的输入输出：
- 函数签名：参数类型 + 返回类型
- 副作用：是否修改全局状态？是否写文件/DB？
- 异常：哪些情况抛什么异常？

### 2. 列举测试用例
按等价类划分 + 边界值分析：

```
正常用例：
- 输入：<典型值> → 期望：<典型输出>

边界用例：
- 输入：空集合/空字符串/0/最小值 → 期望：<处理方式>
- 输入：极大值/超长字符串 → 期望：<处理方式>

异常用例：
- 输入：null/None → 期望：抛 ValueError
- 输入：非法类型 → 期望：抛 TypeError
- 输入：格式错误 → 期望：抛具体异常

组合用例：
- 多参数组合的代表性场景
```

### 3. 实现代码
按接口和测试用例实现，每写完一段就跑测试。

### 4. 验证 + 补测试
- 跑所有测试用例，确保通过
- 思考遗漏的场景，补充测试
- 覆盖率检查（关键分支必须覆盖）

## 测试用例设计原则

### 等价类划分
- 有效等价类：合法输入的代表值
- 无效等价类：非法输入的代表值
- 每个等价类至少 1 个用例

### 边界值
- min-1, min, min+1
- max-1, max, max+1
- 空集合 / 单元素 / 多元素
- 0 / 负数 / 正数

### 状态迁移
- 状态机：覆盖所有状态转移路径
- 例：pending → running → completed / failed

## 测试金字塔

```
       /\
      /E2E\        少量，验证关键链路
     /------\
    /集成测试\      适量，验证模块协作
   /----------\
  /  单元测试  \    大量，验证单个函数/类
 /--------------\
```

- 单元测试：快、独立、覆盖率高
- 集成测试：验证模块间接口
- E2E：验证用户场景，慢但真实

## Mock 原则

- Mock 外部依赖（DB/API/文件系统），不 Mock 被测代码
- Mock 行为要贴近真实（返回结构、异常类型）
- 验证 Mock 被调用的参数和次数
- 避免"Mock 自己"——测试就失去意义

## 输出格式

写代码前输出：

```
【接口】
def func_name(param1: type, param2: type) -> return_type:
    """<一句话描述>"""

【测试用例】
1. 正常：func(a, b) == expected
2. 边界：func("", 0) raises ValueError
3. 异常：func(None, b) raises TypeError
4. ...

【实现要点】
- <关键算法/数据结构>
- <边界处理>
- <异常处理>
```

## 禁止行为

- 禁止先写实现再补测试（容易漏边界）
- 禁止测试覆盖率 100% 但全是无效断言
- 禁止测试依赖执行顺序
- 禁止 Mock 一切导致测试无法发现真实问题
- 禁止不跑失败用例就提交
