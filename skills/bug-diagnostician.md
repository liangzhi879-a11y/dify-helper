# Skill: Bug Diagnostician

按科学调试流程定位 bug，禁止盲目猜测乱改。先复现→假设→验证，再修复。

## 调试流程（必须按顺序）

### 1. 复现问题
- 拿到具体的复现步骤、输入数据、期望结果、实际结果
- 如无法稳定复现，先尝试构造能复现的最小用例
- **禁止在没有稳定复现路径时开始修 bug**

### 2. 收集信息
- 错误堆栈完整内容
- 触发时机（首次/偶发/特定输入）
- 环境信息（版本/配置/依赖）
- 最近改动（git log/diff）

### 3. 形成假设
基于堆栈和信息，列出 2-3 个可能的根本原因，按可能性排序：

```
假设 1（最可能）：<描述>
  依据：<证据>
  验证方式：<如何确认>
假设 2：<描述>
  依据：<证据>
  验证方式：<如何确认>
假设 3：<描述>
  依据：<证据>
  验证方式：<如何确认>
```

### 4. 验证假设
- 从最可能的假设开始验证
- 用最小侵入方式插桩：加日志/print/断点
- **禁止直接改代码验证**——改了就回不去原状态
- 验证后明确结论：假设成立/不成立

### 5. 定位根因
- 假设验证通过后，精确定位到文件+行号+语句
- 区分"表象"和"根因"：例如 OOM 是表象，内存泄漏是根因

### 6. 修复
- 最小化修改：只改根因，不重构
- 修复后立即用第 1 步的复现用例验证
- 思考是否有同类问题需要一并修复

### 7. 回归测试
- 跑相关测试套件
- 思考修复是否引入新问题
- 加一个针对该 bug 的回归测试用例

## 输出格式

```
【症状】<一句话描述问题>
【复现步骤】
1. ...
2. ...
【期望】<应该的行为>
【实际】<实际行为 + 完整错误>

【假设列表】
1. <假设1>（可能性：高/中/低）
2. <假设2>（可能性：高/中/低）

【验证过程】
- 验证假设1：<方法> → <结果>
- ...

【根因】<精确定位：file:line + 原因解释>
【修复】<具体改动>
【验证】<回归测试结果>
```

## 禁止行为

- 禁止不复现就改代码
- 禁止一次改多处再验证（无法判断哪个改动起作用）
- 禁止只修表象不修根因（如 catch 异常让其不报错）
- 禁止不写回归测试就关闭 bug
- 禁止盲目重试（"再试一次看会不会好"）

## 常见陷阱

- **缓存问题**：清缓存再复现
- **环境差异**：本地能跑线上报错，检查依赖版本/环境变量
- **时区/时间**：UTC vs 本地时间
- **并发**：单线程能复现 vs 多线程偶发
- **数据依赖**：特定数据状态才触发
